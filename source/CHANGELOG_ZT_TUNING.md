# ZeroTrust 접근제어 튜닝 변경 내역

## 목적
담당 사건만 목록에 노출하는 방식 대신, 제로트러스트 시연 폭을 유지하면서 실제 민감정보 접근은 상세 조회 시점에 동적으로 평가하도록 보완했다.

## 변경 사항

1. 사건 목록 최소 정보화
   - `api/resource_handler.py`
   - `/api/resources/cases` 목록 API에서 `description` 컬럼을 조회하지 않도록 수정했다.
   - 프론트 호환을 위해 `description` 키는 유지하되, 실제 상세 설명 대신 안내 문구를 반환한다.
   - 상세 내용은 `/api/resources/cases/<id>` 접근 평가 이후 마스킹 정책에 따라 반환된다.

2. 비담당 사건 접근 위험 점수 반영
   - `core/access_evaluator.py`
   - `core/scoring_engine.py`
   - 담당 사건이 아닌 사건에 접근하면 `BEH_UNAUTHORIZED_ACCESS` 기본 +10점을 행위 위험도에 반영한다.
   - 4~5등급 비담당 사건이면 `BEH_HIGH_SENS_UNASSIGNED` 기본 +5점을 추가 반영한다.
   - 단일 접근은 실수 가능성을 고려해 소폭 가산만 수행하고, 기존 의사결정 엔진이 점수에 따라 조회 허용/재인증/승인/차단을 결정한다.

3. 짧은 시간 내 산발적 접근 기준 단계화
   - `core/anomaly_service.py`
   - `core/scoring_engine.py`
   - 5분 내 5회 이상 접근: 주의, 기본 +10점
   - 5분 내 10회 이상 접근: 위험, 기본 +20점
   - 기존 5분 내 10회 단일 기준보다 시연에서 이상 행동 흐름을 더 명확히 보여줄 수 있다.

4. 정책 임계값 마이그레이션 추가
   - `migrations/020_zerotrust_access_tuning.sql`
   - 신규 정책값을 `policy_thresholds`에 추가/갱신한다.

## 적용 방법

기존 DB를 사용하는 경우 다음을 실행한다.

```bash
python scripts/run_migrations.py
```

새 환경에서는 일반 초기화 절차대로 마이그레이션을 실행하면 020까지 자동 반영된다.

## 보안 참고

배포 ZIP에서는 `.env`의 실제 SECRET_KEY와 DB 접속 정보를 제거하고 `.env.example` 기반 템플릿으로 대체했다. 실행 전 환경에 맞게 `.env`를 채워야 한다.


## 추가 UI 개선 — 비담당 사건 목록 마스킹
- 사건 목록은 전체 리스팅을 유지
- 담당 사건은 기존 컬러 등급 배지와 기본 메타데이터/설명 표시
- 비담당 사건은 카드 전체를 흑백 톤으로 표시하고 제목, 사건번호, 등급 배지, 자료유형, 부서, 설명을 블러 처리
- 비담당 사건도 클릭 가능하게 유지하여 반복 접근/의도적 탐색 시 제로트러스트 위험 점수 탐지가 가능
- 목록 API 응답에 `is_assigned_case` 필드 추가
- 비담당 사건 목록 데이터는 브라우저에도 최소 정보(비공개/비담당 사건)만 전달


## 추가 UI 개선 — 비담당 사건 클릭 전 경고 모달
- 비담당 사건 카드를 클릭하면 즉시 상세 API를 호출하지 않고 경고 모달을 먼저 표시
- 사용자가 취소하면 위험 점수 증가 없이 목록에 머무름
- 사용자가 "접근 평가 진행"을 누른 경우에만 상세 접근 API 호출
- 경고 문구에 담당 사건 미등록, 감사 로그 기록 가능성, 반복 접근 시 계정 제한/차단 가능성을 명시


## 추가 기능 — 담당 사건 등록 요청 플로우
- 비담당 사건 경고 모달의 버튼을 `접근 평가 진행`에서 `담당 사건 등록 요청`으로 변경
- 버튼 클릭 시 `/api/resources/cases/:id/assignment-request`로 담당 사건 등록 요청 생성
- 일반 사용자의 요청은 관리자에게, 관리자/부관리자의 요청은 자기승인 방지를 위해 부관리자에게 전달
- 관리자가 요청자 OTP 인증을 요구할 수 있는 API 추가
- 요청자가 OTP 인증을 완료하면 관리자가 최종 승인하여 `users.assigned_cases`에 사건 ID 추가
- 담당 사건으로 승인된 뒤 사건 목록 재조회 시 컬러/마스킹 해제 구조로 동작
- 마이그레이션 `021_case_assignment_requests.sql` 추가


## 버그 수정 — 사건 목록 담당 여부 판단
- JWT payload에 `assigned_cases`가 없을 수 있어 사건 목록에서 모든 사건이 비담당으로 표시되던 문제 수정
- `CaseListHandler`가 DB의 최신 `users.assigned_cases`를 직접 조회해 담당/비담당을 판단하도록 변경
- 담당 사건 등록 승인 후 목록 재조회 시 컬러/마스킹 해제가 즉시 반영되도록 보완


## 버그 수정/기능 보완 — 관리자 패널 담당 사건 등록 요청 탭
- 관리자 패널에 `담당 사건 등록 요청` 탭 추가
- 일반 사용자 요청은 관리자 계정에서, 관리자/부관리자 요청은 부관리자 계정에서 조회되도록 표시
- 요청 건별 `요청자 OTP 인증 요구`, `최종 승인 / 담당 등록`, `반려`, `새로고침` 버튼 추가
- 요청자가 다시 비담당 사건 등록 요청 모달을 열면 `otp_required` 상태에서 OTP 입력/인증 가능
- OTP 인증 완료 후 관리자가 최종 승인하면 `users.assigned_cases`에 사건 ID가 추가되어 담당 사건으로 전환됨


## 최종 수정 — 담당 사건 등록 OTP 흐름/마스킹 UI 보완
- 관리자 패널의 `최종 승인 / 담당 등록` 버튼을 요청자 OTP 인증 완료(`otp_verified`) 후에만 활성화
- 관리자 패널의 설명 박스 제거
- 요청자 화면에 `내 담당 사건 등록 요청` 상태 조회 API를 5초 주기로 폴링
- 관리자가 OTP 인증을 요구하면 요청자 사건 목록 화면 상단에 OTP 재인증 UI 자동 표시
- 요청자 OTP 인증 완료 후 관리자 패널에서 최종 승인 가능
- 비담당 사건 카드에서 사건명은 보이도록 변경하고, 사건번호/등급/자료유형/부서/세부 설명만 흑백/블러 처리
- 비담당 사건 목록 API도 사건명은 내려주되 나머지 민감 메타데이터는 최소화 유지


## 최종 수정 — OTP 요구 실시간 팝업 및 승인 후 자동 마스킹 해제
- 요청자 화면이 담당 사건 등록 요청 상태를 1초 주기로 폴링
- 관리자가 OTP 인증 요구를 누르면 요청자 화면에 새로고침 없이 OTP 재인증 모달 자동 표시
- 기존 상단 배너 방식은 제거하고 팝업 모달 방식으로 변경
- OTP 인증 완료 전까지 관리자 패널의 `최종 승인 / 담당 등록` 버튼 비활성화 유지
- 관리자 최종 승인 후 요청자 화면에서 사건 목록을 자동 재조회해 컬러/마스킹 해제 반영


## UI 수정 — OTP 재인증 모달 레이어 우선순위
- 관리자가 OTP 인증 요구 시 기존 `담당 사건 등록 요청` 모달을 자동으로 닫음
- OTP 재인증 모달 z-index를 `z-[80]`으로 올려 다른 모달 위에 표시
- 크롬 사용자 화면에서 새로고침 없이 OTP 재인증 창이 전면에 뜨도록 보완
